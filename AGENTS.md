# AGENTS.md

## 项目级硬约束

1. 本项目当前以本机 `D:\Workspace\agent-platform` 开发和验证为主；Git 写操作默认禁止，只有用户明确授权时才可执行。
2. 如果没有用户明确授权，只允许执行 Git 只读检查，例如 `git status`、`git diff`、`git log`。2026-07-16 用户已明确授权本次初始化仓库、提交并推送到 `https://github.com/atlantis68/agent-platform.git`。
3. 代码注释需要尽量完整，便于后续人工阅读和交接；公共接口、核心流程、风险边界、协议字段、非显而易见的设计决策必须写清楚。
4. 所有代码备注必须使用中文，包括 JavaDoc、Python docstring、行注释、块注释和测试辅助说明；可以保留 Agent、Runtime、DTO、HTTP、SSE、MCP 等业界通用英文技术名词，但叙述句必须使用中文。
5. 注释要解释“为什么这样做”和“边界是什么”，不要堆砌“给变量赋值”这类空洞注释。
6. 每个阶段都必须有成果展示和交付物，不能只交付抽象设计或零散代码。
7. 项目过程中遇到的问题、根因、修复方式、验证方式和预防措施，必须沉淀到项目专属 Skill：`agent-platform-project-governance`。
8. 后续开发阶段交互时，必须详细说明“为什么这样做”和“做了什么”，便于后续归档、审查和人工接手。
9. 过程中如果出现必须人工介入的事项，必须立即停下来反馈阻塞点、影响范围、需要用户确认的内容；确认清楚前不得继续执行相关主链路。

## 交互与归档规则

后续开发阶段的每次关键交互，需要尽量包含：

1. 当前要解决的问题或阶段目标。
2. 为什么选择当前做法，而不是只写结论。
3. 已经做了什么，包括创建/修改的关键文件、运行的关键命令和验证结果。
4. 尚未做什么，以及原因。
5. 下一步准备做什么。

如果信息已经写入 `progress.md`、阶段展示文档或 Skill 问题日志，最终回复中也需要指出归档位置。

## 人工介入规则

遇到以下情况必须暂停相关主链路，并向用户确认：

1. 需求、范围、接口、技术路线存在会影响后续实现的歧义。
2. 需要执行高风险操作，例如 Git 写操作、删除/覆盖文件、数据库变更、外部系统写入、CLI 危险命令。
3. 缺少必要凭据、环境、模型配置、数据库连接或第三方服务授权。
4. 测试、构建、启动连续失败，且继续尝试可能扩大影响或偏离原方案。
5. 发现既有规则互相冲突，或用户新要求与已落地规则不一致。

## 阶段交付规则

每个阶段至少包含：

1. 阶段目标和范围。
2. 可运行或可审查的成果展示。
3. 交付物清单。
4. 验收标准。
5. 已知限制。
6. 下一阶段入口条件。

## 问题沉淀规则

遇到以下任一情况，必须更新 `D:\Workspace\agent-platform\.agents\skills\agent-platform-project-governance\references\problem-log.md`：

- 架构或协议决策发生调整。
- 构建、启动、测试、依赖、环境、编码、权限或工具调用出现异常。
- 出现误判、返工、假阳性扫描、工具脚本失败。
- 发现需要长期遵守的新约束或新经验。

记录格式必须包含：时间、阶段、现象、根因、修复、验证、预防措施、关联文件。
