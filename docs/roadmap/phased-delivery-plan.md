# 企业内部通用 Agent 平台分阶段交付路线

## 总体原则

平台按“协议先行、闭环优先、能力递进、治理兜底”的顺序建设。每个阶段必须交付可展示成果，不能只交付代码或抽象设计。

## 阶段总览

| 阶段 | 名称 | 目标 | 成果展示 | 交付物 | 验收标准 |
|---|---|---|---|---|---|
| 0 | 架构与协议基线 | 固化混合架构、运行协议、事件协议、工具协议与阶段边界 | 架构图、运行时序图、事件样例、阶段验收表 | 架构文档、Runtime 协议、交付路线、阶段 0 展示稿 | 团队能据此拆分前后端、控制面、Runtime 工作 |
| 1 | 最小 Agent Runtime | 跑通创建 Agent、发起运行、模型返回、短期记忆、基础审计 | Web 页面输入问题后看到流式回答和运行记录 | Java 控制面 v1、Python Runtime v1、前端运行台 v1 | 一次 Agent Run 从创建到完成全链路可追踪 |
| 2 | 工具系统与 MCP v1 | Agent 可调用平台工具和 MCP tools | Agent 调用一个 HTTP 工具和一个 MCP 工具并展示审计日志 | Tool Registry、HTTP Tool Adapter、MCP Client、工具调用协议、风险分级 | 工具输入输出、耗时、错误、权限结果可审计 |
| 3 | RAG 与企业知识库 | Agent 可基于企业文档回答并给出来源 | 上传文档后提问，回答显示引用来源 | 文档导入、切分、Embedding、检索、Rerank、知识权限 | 答案包含来源，越权用户无法检索受限知识 |
| 4 | 长期记忆 | Agent 可跨会话使用授权记忆 | 用户偏好跨会话生效，并可查看/删除 | 记忆模型、写入策略、检索策略、冲突处理、管理界面 | 记忆写入有来源，用户可管理，冲突可解释 |
| 5 | Skill 能力体系 | 将可复用流程沉淀为版本化 Skill | 同一 Agent 按任务选择不同 Skill 执行 | Skill Manifest、版本、依赖工具、内部市场、评测集 | Skill 可发布、启停、回滚、评测 |
| 6 | CLI 与沙箱执行 | Agent 可在隔离环境执行受控命令 | 在沙箱中执行只读命令并展示审批拦截 | 工作区隔离、命令策略、资源限制、审批、脱敏 | 危险命令被拒绝或进入人工审批 |
| 7 | 企业治理与可靠性 | 平台具备上线级权限、审批、可靠执行能力 | 高风险工具触发审批，中断后可恢复 | RBAC、Agent 身份、HITL、Guardrails、持久化工作流、多 Agent | 长任务可恢复，所有高风险动作可追责 |
| 8 | 观测、评测与运营 | 平台可持续运营和质量改进 | 运营看板展示成功率、成本、质量趋势 | Trace、回放、成本、评测、回归测试、灰度发布 | 模型/Skill 升级前后可量化对比 |

## 阶段推进规则

1. 每阶段开始前写明目标、范围、接口变更和验收标准。
2. 每阶段中途保留可演示的主链路，不把演示留到最后一天。
3. 每阶段结束时输出阶段展示文档，说明“能看什么、怎么验收、哪里有限制”。
4. 后一阶段不得破坏前一阶段的演示闭环。
5. 风险能力按默认关闭处理，尤其是写操作工具、CLI、外部网络与敏感数据访问。

## 阶段 2 执行边界

阶段 2 已完成实施计划，但尚未开始编码。该阶段只做工具系统和 MCP tools 的最小闭环：

1. 先做 Tool Registry、工具元数据、输入 schema、风险等级和审计事件。
2. 只接入一个低风险 HTTP 工具和一个 MCP tool，证明协议闭环成立。
3. 暂不做 MCP resources、MCP prompts、RAG、长期记忆、Skill 运行时和 CLI 沙箱。
4. 高风险写操作默认拒绝；如需开放，必须进入后续治理阶段。

## 阶段 2 推荐落地顺序

阶段 2 采用“控制面注册表先行、Runtime 执行闭环跟进、审计展示收口”的顺序，原因是工具能力属于高风险扩展点，必须先固定元数据、输入约束和风险等级，再允许 Runtime 执行。

1. **Tool Registry：** 控制面提供工具列表接口，返回工具名称、描述、来源类型、输入 schema 和风险等级。
2. **Runtime 工具执行：** Runtime 根据用户输入触发低风险 `http_echo` 或 `mcp_local_time`，并追加工具审计事件。
3. **审计与展示：** 控制面从 Runtime 拉取工具事件，更新 `usage.toolCalls`，并通过审计接口和 SSE 展示。
4. **阶段交付：** 输出 `docs/delivery/phase-2-showcase.md`，记录验收命令、结果、已知限制和阶段 3 入口条件。

阶段 2 默认不阻塞于真实业务配置；如果用户暂不提供 HTTP API 或 MCP server，就使用 demo 工具完成平台级闭环。

## 推荐里程碑节奏

| 里程碑 | 覆盖阶段 | 结果 |
|---|---|---|
| M0 | 阶段 0 | 架构和协议可评审 |
| M1 | 阶段 1-2 | Agent 能对话并调用工具 |
| M2 | 阶段 3-4 | Agent 具备企业知识和记忆 |
| M3 | 阶段 5-6 | Agent 能复用技能并执行受控自动化 |
| M4 | 阶段 7-8 | 平台达到企业上线治理要求 |
